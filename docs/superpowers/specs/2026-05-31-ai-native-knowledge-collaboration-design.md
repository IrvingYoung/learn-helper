# LLM Wiki AI-Native 知识协同设计

## 背景

### 当前产品状态

LLM Wiki 已经完成了一期核心功能（聊天→Plan→确认→写入→知识树），二期功能也在快速推进中：
- `propose_plan` 扩展了 `outline` / `phases` / 分参数字段
- `ExecOutline` 支持递归创建骨架页面
- `PlanPreview` 支持大纲树可视化 + 阶段进度
- `OperationQueue` 支持待确认操作列表 + 执行结果卡片
- `KnowledgeTree` 支持右键菜单、拖拽、内联重命名
- `PageViewer` 支持 Tab 切换（内容/操作）
- Legacy pending_actions 代码已清理

### 核心痛点

产品演化过程中暴露了三个交织的问题：

1. **用户 vs AI 的所有权矛盾**
   - AI 被设定为"知识库管理员"，主动规划结构、创作内容
   - 用户对知识树结构和内容质量有自己的标准
   - 结果：AI 提议 → 用户不满意 → 聊天改 → 再提议 → 再改，拉锯

2. **审批前置导致摩擦过高**
   - 每次写入都是一笔交易——AI 提案，用户逐个审 action，确认后再执行
   - 用户的核心操作变成了"审批"而不是"学习"
   - PlanPreview 展示 action 列表（create_page / update_page / link_pages），用户审的是工程细节而不是内容本身

3. **节奏控制不匹配学习场景**
   - AI 一次性完成大量操作，用户来不及细审
   - "记东西"和"学东西"没有分开——学的时候被审批打断

### 根因

**当前协作模型是 AI 主导的，但学习型知识库的用户天然应该是主导者。**

```
当前模式：AI 写 → 用户审 → 改了再写 → 用户再审（拉锯）
理想模式：用户定方向 → AI 填充 → 用户微调 → 确认（顺畅）
```

## 新协作模型

### 三条原则

1. **你决定是否记，AI 决定记什么**
2. **聊天只聊，内容放内容区**
3. **审阅审内容本身，不审工程细节**

### 协作分界线

| 用户做的 | AI 做的 |
|---|---|
| 说"记下来"、"改这里"、"算了" | 写内容、选位置、建链接 |
| 审阅内容本身 | 根据校准方向产出内容 |
| 决定下一步学什么 | 提议学习路径/大纲 |
| 直接调整树结构（拖拽/右键/重命名） | 建议结构调整（聊天中提出） |
| 创建顶层页面（右键添加） | 在用户创建页面上生成大纲 |

### 核心变化：从"审批前置"到"迭代后置"

| 维度 | 旧模式 | 新模式 |
|---|---|---|
| 操作心态 | 每次写入是一笔交易 | 内容是活的，可迭代 |
| 确认时机 | 写之前先审批 | 写之后用户可以改 |
| 聊天角色 | 操作入口 | 学习和讨论入口 |
| 内容展示 | PlanPreview action 列表 | 页面内内容 + 确认条 |
| 调整方式 | 拒绝 Plan → 重新提案 | 聊天里说"这里改一下" |
| 操作粒度 | 离散的、原子级的 | 连续的、可迭代的 |

## 交互设计

### 场景一：构建大纲

用户创建一个页面后，让 AI 生成学习大纲和目录结构。

```
用户在左栏右键新建「Go」页面（空内容）
  ↓
用户：帮我生成 Go 的学习大纲
  ↓
AI 在右栏展示大纲预览：
┌─ 大纲预览 ──────────────────────────┐
│  # Go 学习路线                        │
│                                      │
│  第一阶段：Go 基础语法                │
│    变量与类型                        │
│    控制流                            │
│    函数                              │
│  第二阶段：并发编程                   │
│    goroutine                         │
│    channel                           │
│  第三阶段：Web 开发                   │
│  第四阶段：项目实战                   │
│                                      │
│  [确认]                               │
└──────────────────────────────────────┘

用户在聊天中调整结构（右栏实时更新）：
用户：把并发放到 Web 开发后面
  → 右栏大纲更新
用户：差不多了
  → 点确认

确认后 AI 做两件事：
1. 将大纲内容写入「Go」页面
2. 创建子目录（空骨架页面）

知识树：
Go (5)
├── Go 基础语法 (空)
├── 并发编程 (空)
├── Web 开发 (空)
└── 项目实战 (空)
```

**设计要点：**
- 大纲 = 页面内容 + 子目录，不需要额外概念
- 学习路线 = 知识树（顶层节点按学习阶段排列）
- 调整在聊天中说，右栏实时更新
- 确认只有"确认"按钮，没有"重写"按钮（重写就是自然对话）

### 场景二：学习与内容沉淀

用户沿着大纲学习，有收获时让 AI 写入知识库。

```
知识树上用户点击「Go 基础语法」
  ↓
用户：先学基础语法
  ↓
AI 在聊天中讲解、用户追问、讨论
  ↓
用户有收获 → "记下来"
  ↓
AI 在聊天中问 1-2 个校准问题：
  "关于变量声明，你想注重什么角度？
   A) 和 Python 的对比
   B) 常见踩坑经验
   C) 底层实现原理"
  ↓
用户选择后，AI 在右栏页面内写入内容：

┌─ [有未确认的内容] [确认] ──────────┐
│                                     │
│ # Go 基础语法                       │
│                                     │
│ Go 的变量声明有 var 和 := 两种方式… │
│ ……                                 │
└─────────────────────────────────────┘

聊天中：
AI：已经写入了，你看看

用户看后：
用户确认 → 点"确认"按钮
或用户不满意 → "把类型推断的部分再展开说一下"
  → AI 迭代更新页面内容
```

**设计要点：**
- 内容展示在"页面内"（右栏），不是聊天里
- 确认条在页面顶部，而不是单独的 PlanPreview 面板
- 调整通过自然对话，不需要"拒绝 Plan → 重新提案"
- 校准追问在聊天中进行，不占页面空间

### 场景三：树直接操作

用户在日常浏览和知识整理时直接操作树。

| 操作 | 方式 | 确认 |
|---|---|---|
| 添加子页面 | 右键菜单 / + 按钮 | 直接执行 |
| 重命名 | 双击 / 右键菜单 | 直接执行 |
| 移动 | 拖拽 / 右键菜单 | 直接执行 |
| 删除页面 | 右键菜单 | 需确认（右栏确认条） |

AI 也可以主动建议结构调整（在聊天中提出），由用户决定。

## 关键设计决策

### 1. 没有"大纲"和"页面"的区分

**树 = 学习路线 = 知识库。** 每个节点就是一篇页面，空节点是未学的内容，已填充的是已学的部分。用户只需看谁还是空的就知道下一步学什么。

### 2. 内容预览在页面内，不在聊天里

聊天只承载对话本身，内容展示在右栏对应页面内。避免聊天被大段内容撑爆，也让内容出现在它该出现的地方。

### 3. 确认条替代 PlanPreview

AI 写入内容后，在页面顶部展示一个轻量确认条：

```
┌─ [有未确认的内容] [确认] ───────┐
```

没有"重写"按钮——需要调整就在聊天里说。没有 action 列表——用户审内容本身，不审工程细节。

### 4. PlanPreview 的保留和退役

| 场景 | 组件 |
|---|---|
| AI 提案大纲（树结构） | PlanPreview 的 outline 树视图 |
| AI 写内容（"记下来"） | 页面内确认条（不经过 PlanPreview） |
| 执行结果 | 不再需要独立展示——内容写进页面本身就是结果 |

PlanPreview 保留大纲树展示功能，内容部分直接退役。
OperationQueue 也退役——没有"待确认操作Tab"，确认在页面内完成。

### 5. 未确认内容的处理

AI 写入后未确认的内容会保留在页面上，保持"未确认"状态。页面顶部的确认条一直存在，直到用户确认或说"算了"。不会自动确认，不会消失。

## AI 角色重写

### 旧角色：知识库管理员

```diff
- 你是 LLM Wiki 的知识库维护者。你的职责是管理知识树，
- 包括创建、更新、删除页面和建立页面间的链接。
```

### 新角色：学习伙伴 + 内容创作者

系统需要引导 AI 表现出以下行为模式：

1. **先校准再输出** — 写内容前先问 1-2 个问题了解用户的需求、背景、关注的深度
2. **记下来的内容要自然** — 内容风格贴近对话上下文，不是百科搬砖
3. **大纲就是页面** — 生成大纲时直接在页面上写内容、建子目录
4. **主动建议** — 发现知识体系不完整时在聊天中提出建议
5. **接受迭代** — 用户说"改这里"时，不是重新走 Plan，而是直接在页面上更新

（具体的 system prompt 文本在实现阶段细化）

## 影响范围

### 前端改动

| 文件 | 变更类型 | 说明 |
|---|---|---|
| `PageViewer.tsx` | 大改 | 移除 Tab 布局，直接展示页面内容 + 确认条 |
| `PlanPreview.tsx` | 中改 | 保留 outline 树展示，移除 action 确认 UI |
| `OperationQueue.tsx` | 删除 | 不再需要 |
| `ChatPanel.tsx` | 小改 | 移除待确认操作相关逻辑 |
| `KnowledgeTree.tsx` | 无改 | 已有直接操作能力 |
| `api.ts` | 小改 | 新增 "确认内容" API |

### 后端改动

| 文件 | 变更类型 | 说明 |
|---|---|---|
| `ai/provider.go` | 中改 | System prompt 重写 |
| `handler/ai.go` | 小改 | 移除旧的 Plan 确认逻辑 |
| `engine/engine.go` | 小改 | 支持 "auto_execute" 模式（不确认直接写入）|

### 已实现功能的复用

| 已有功能 | 如何处理 |
|---|---|
| `ExecOutline` | 保留，确认大纲后执行骨架创建 |
| `propose_plan` outline 字段 | 保留，用于大纲提案 |
| `TreeNodeMenu` / 右键菜单 | 保留，直接操作能力 |
| `KnowledgeTree` 拖拽功能 | 保留 |
| Plan 执行结果 | 简化——成功则内容已写入页面，失败则在聊天报错 |

## 实现顺序

### Phase 1：System Prompt 重写 + 确认条（最小改动，最大效果）

重写 `buildWikiMaintainerPrompt`，让 AI 行为从"管理员"变为"学习伙伴 + 内容创作者"。同时在 PageViewer 中增加页面内确认条。这个阶段不改 Plan 系统底层。

### Phase 2：内容确认迁移到页面内

确认不再经过 PlanPreview——AI 写入内容后直接在页面内展示 + 确认条。学习流程改为"说记下来 → AI 校准 → 写入页面 → 确认条"。

### Phase 3：清理退役组件

删除 OperationQueue、移除 PageViewer Tab 布局、清理 Plan 确认相关旧代码。

### Phase 4：剩余功能

webtools、theme-refactor、materialized-path 等独立功能按需推进。

## 成功指标

- 从"提出需求"到"看到内容"的操作步骤减少 ≥ 50%
- 用户不再审 action 列表，只审内容本身
- 树直接操作（拖拽/右键）的占比 > 直接操作总量的 60%
- 用户在聊天中说"改这里"而不是"拒绝 Plan"的频率 = 100%
