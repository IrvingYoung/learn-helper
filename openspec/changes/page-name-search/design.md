## Context

用户需要在知识树中按名称快速定位 wiki 页面。目前知识树组件（KnowledgeTree）从父组件接收完整的 `WikiTreeNode[]` 树形数据，没有搜索/过滤功能。

这是纯前端改动：
- 后端已有 `GET /api/wiki` 返回完整知识树（包含所有 page 的 title），无需新增 API
- 搜索只匹配页面名称（title），不涉及页面内容
- 知识树组件是受控组件，数据通过 props 传入

## Goals / Non-Goals

**Goals:**
- 在知识树顶部增加一个搜索输入框
- 输入关键词时实时过滤树节点，只显示 title 包含关键词的节点
- 搜索时保持祖先路径可见（方便用户知道匹配节点在树中的位置）
- 无搜索关键词时恢复完整树展示
- 搜索框有清空按钮

**Non-Goals:**
- 不搜索页面内容（body/content）
- 不做后端搜索、不做模糊匹配或拼音搜索
- 不修改后端 API
- 不新增外部依赖

## Decisions

1. **客户端过滤而非新增后端搜索 API**
   - 当前 `GET /api/wiki` 已返回所有 page 的 title，数据量小（知识库规模通常在数百以内），客户端过滤足够快
   - 避免引入后端复杂度

2. **匹配时显示祖先路径而非打平列表**
   - 用户在树中浏览时需要上下文，知道匹配页面属于哪个分支
   - 做法：过滤时保留匹配节点及其所有祖先节点，不匹配的兄弟/子树隐藏

3. **过滤条件：title 包含搜索关键词（大小写不敏感）**
   - 最简单的子串匹配，符合用户对「按名称搜索」的直觉预期

4. **搜索框放在知识树顶部、"知识库"标题下方**
   - 符合常见 UI 模式（如文件管理器的过滤框）
   - 不影响知识树的拖拽、右键菜单等现有功能

## Risks / Trade-offs

- [搜索性能] 树过滤每次按键都重新计算 → 但知识库规模小（<1000 节点），实时计算无压力
- [UI 一致性] 搜索状态和展开/折叠状态可能互相干扰 → 搜索时自动展开所有祖先节点，清空搜索后恢复用户的展开/折叠状态
- [体验] 搜索时禁用拖拽和右键菜单 → 搜索状态和操作状态隔离，避免混淆
